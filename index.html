TypeScript

Typed superset of JavaScript. The interface question is worth the most marks of any single question — learn it precisely.

📌 Version used in this module

TypeScript 5.x (the L30 slide cites v6.0.3). The syntax in these notes — interfaces, union/literal types, optional ?, generics — has been stable for years and is identical across TS 4 / 5 / 6, so nothing version-specific can trip you up here.

What & why

  • Open-source, released October 2012 by Microsoft.
  • A strict superset of JavaScript — every valid .js file is valid TypeScript. It adds optional static typing on top.
  • Cannot run in a browser directly — it is transpiled (compiled) to plain JavaScript by the tsc compiler.
  • Designed for large-scale applications where type errors caught at compile time save you from runtime bugs.
  • Files end in .ts (or .tsx for files containing JSX). Compiler options live in tsconfig.json. It is transpiled to JavaScript with the tsc compiler.

Implicit vs explicit typing

💡 Key distinction

Implicit (inferred): TypeScript guesses the type from the value you assign. Explicit: you write : type yourself. Once a type is fixed (either way) you cannot assign a different type.

// IMPLICIT — inferred as number from the value 23
let a = 23;
a = "Hi";            // ❌ ERROR: Type 'string' is not assignable to type 'number'

// EXPLICIT annotations
let name: string = "COS216";
let count: number = 0;
let active: boolean = true;

// 'any' opts out of checking — legal but defeats the point of TS
let b: any = 23;
b = "Hi";            // ✅ allowed because type is 'any'
What this does, line by line
  • let a = 23; — you never wrote a type, so TypeScript infers one from the value 23. It silently decides "a is a number" forever.
  • a = "Hi"; — now you try to put text into that number variable, so the compiler stops you with an error. This is the whole point of TS: the mistake is caught before the program runs.
  • let name: string = "COS216"; (and the two lines below it) — here you explicitly write the type after a colon. name can only ever hold a string, count only a number, active only true/false.
  • let b: any = 23; then b = "Hi"; — declaring any turns checking off for that variable, so switching it from a number to a string is allowed. It works, but you've thrown away the safety, so only use any when you genuinely don't know the type.
⚠️ Mistakes to avoid
  • Using any everywhere — it silently disables the safety the exam is testing for. Only worth marks when the spec literally says "can be anything".
  • Confusing type annotation (let x: number) with type assertion (let x = y as number).
  • Writing String/Number/Boolean (capitalised wrapper objects) instead of the lowercase primitives string/number/boolean.

Custom (union) types & literal types

💡 The single most exam-relevant feature

A union (|) means "one of these". A string-literal union restricts a value to an exact set of strings — this is how the exam expects you to model things like assessment type or period.

type Style = 'italic' | 'bold' | 'normal';
let font: Style;
font = 'underline';   // ❌ ERROR — only the three listed strings are allowed
font = 'bold';        // ✅

// union of two primitive types — "either a string OR a number"
let code: string | number;
code = "COS216";       // ✅
code = 216;            // ✅
What this does, line by line
  • type Style = 'italic' | 'bold' | 'normal'; — creates your own named type. The | means "OR", so a Style value is allowed to be only one of those three exact words.
  • let font: Style; — declares font as that type, so the compiler now polices what you put in it.
  • font = 'underline'; — rejected, because 'underline' isn't in the allowed list of three.
  • font = 'bold'; — accepted, because it matches one of the allowed words.
  • let code: string | number; — a union of two types rather than words: code may hold either a string or a number, which is exactly how you model "can be alphanumeric or numeric" in the interface question.

Enums

An enum names a set of constants. Numeric enums auto-increment from 0; string enums keep their string value.

enum PizzaTopping { TOMATO, BBQ, NONE, CREAM }  // 0,1,2,3
enum PizzaSize    { S = 's', M = 'm', L = 'l', XL = 'xl' }
let t = PizzaTopping.BBQ;   // 1
What this does, line by line
  • enum PizzaTopping { TOMATO, BBQ, NONE, CREAM } — a numeric enum. You didn't give values, so TypeScript numbers them automatically starting at 0: TOMATO=0, BBQ=1, NONE=2, CREAM=3.
  • enum PizzaSize { S = 's', ... } — a string enum: here you set each member's value yourself, so PizzaSize.M is the string 'm'.
  • let t = PizzaTopping.BBQ; — reads a member by its name; t ends up holding 1 (BBQ's auto-assigned number).
💡 Enum vs string-literal union

Both restrict to a set of values. Prefer a string-literal union when the spec says a field "can only be 'A' or 'B'" — it is exactly what the markers expect for interface fields. Use an enum when you need named constants used throughout the code.

Interfaces — the big one

🎯 Worth 14 marks in 2021 (Q5)

You will be given a plain-English spec of an object and asked to write a strongly-typed interface. Marks are awarded per correctly-typed field. Memorise the mapping from English → type.

// This is the EXACT 2021 Q5 answer — the Module interface
interface Module {
  code: string | number,             // alphanumeric OR numeric
  name: string,
  description: string,
  prerequisites: string[],            // array of module codes
  assessment: 'SUMMATIVE' | 'FORMATIVE',
  year: number,
  period: 'Q1' | 'Q2' | 'S1' | 'Q3' | 'Q4' | 'S2',
  submin?: number,                    // optional — the '?' is the mark
  active: boolean
}

// using it — every required field must be present and correctly typed
const cos216: Module = {
  code: 216, name: "Concurrent Systems", description: "...",
  prerequisites: ["COS110", "COS212"], assessment: 'SUMMATIVE',
  year: 2, period: 'S1', active: true   // submin omitted — OK, it's optional
};
What this does, field by field

An interface is a shape/contract: it lists what fields an object must have and what type each one is. It produces no real code — it only tells the compiler how to check objects you label as a Module.

  1. interface Module { — starts the contract named Module.
  2. code: string | number, — the field code may be a string or a number (this is how "alphanumeric or numeric" is written).
  3. name: string, and description: string, — plain text fields.
  4. prerequisites: string[], — the [] means "an array (list) of". So this holds many module-code strings, e.g. ["COS110","COS212"]. "A list of…" always becomes an array.
  5. assessment: 'SUMMATIVE' | 'FORMATIVE', — a string-literal union: this field can only be one of those two exact words, nothing else.
  6. year: number, — a plain number (a year is just a number, not a Date).
  7. period: 'Q1' | 'Q2' | ..., — same idea as assessment, but with six allowed values.
  8. submin?: number, — the ? makes this field optional: an object can leave it out and still be a valid Module.
  9. active: boolean — true/false flag. Note: no comma after the last field, then } closes the interface.
  10. The const cos216: Module = { … } block then uses the interface: by labelling the object : Module, the compiler checks every required field is present and correctly typed. submin is left out, which is fine because it was optional.
⚠️ Exact mistakes a real 2021 script made (and lost marks)
  • prerequisites: string instead of string[] — "a list of" always means an array.
  • year: Date() — the spec said "a year", which is a number, not a Date.
  • Forgetting the ? on the optional submin field.
  • Quoting the literal-union values inconsistently, or using = instead of : between field and type.

Classes

class Point {
  // 'public' in the constructor declares AND assigns the field in one line
  constructor(public x: number, public y: number) {}

  add(p: Point): Point {
    return new Point(this.x + p.x, this.y + p.y);
  }
}
const p3 = new Point(0, 10).add(new Point(10, 20)); // {x:10, y:30}
What this does, line by line
  • class Point { — defines a blueprint for "point" objects.
  • constructor(public x: number, public y: number) {} — the constructor runs when you create a point. Writing public in front of each parameter is a TypeScript shortcut that both declares the fields x and y and stores the passed-in values into them — so you don't need separate this.x = x lines.
  • add(p: Point): Point { ... } — a method that takes another point p and returns a new point. The : Point after the brackets says "this method gives back a Point".
  • return new Point(this.x + p.x, this.y + p.y); — builds a brand-new point whose coordinates are the two points added together. this means "the point you called .add on".
  • The last line creates point (0,10), adds point (10,20) to it, and stores the result (10,30) in p3.

Access modifiers: public (default), private (only inside the class), protected (class + subclasses), readonly (assign once).

Typed functions, tuples & generics

// typed function: param types + return type
function pow(x: number, y: number): number { return Math.pow(x, y); }
pow('a', 'b');   // ❌ COMPILE error — caught before it ever runs

// tuple: fixed-length array with a type per position
let row: [string, number] = ["age", 21];

// generic: works with any type T, decided at use-time
class Box<T> { constructor(public value: T) {} }
let n = new Box(216);  // inferred Box<number>
What this does, line by line
  • function pow(x: number, y: number): number — a typed function: both inputs must be numbers and it promises to return a number. So pow('a','b') is rejected at compile time because strings aren't numbers.
  • let row: [string, number] = ["age", 21]; — a tuple: a fixed-length array where each position has its own type. Slot 0 must be a string, slot 1 must be a number.
  • class Box<T> { ... } — a generic class. T is a placeholder for "whatever type you use it with", decided later.
  • let n = new Box(216); — because you passed a number, TypeScript fills in T = number, making this a Box<number>. The same class would work with strings, objects, etc.

Loops: for…in vs for…of

⚠️ Classic trick question

for…in gives keys / indices. for…of gives values.

let list = [4, 5, 6];
for (let i in list)  console.log(i);  // 0 1 2  (indices)
for (let v of list)  console.log(v);  // 4 5 6  (values)
What this does, line by line
  • let list = [4, 5, 6]; — an array with values 4, 5, 6 sitting at positions (indices) 0, 1, 2.
  • for (let i in list) — for…in walks the positions, so i is 0, then 1, then 2.
  • for (let v of list) — for…of walks the values, so v is 4, then 5, then 6.
  • Memory hook: "indices" vs "values of". Mixing these up is a classic exam trap.

Modules: export / import

// point.ts
export class Point { constructor(public x: number) {} }

// main.ts  — note: NO .ts extension in the import path
import { Point } from './point';
What this does, line by line
  • export class Point { ... } — the export keyword makes this class visible to other files. Without it, the class is private to point.ts.
  • import { Point } from './point'; — pulls that exported Point into main.ts so you can use it here. ./point is the file path (same folder); you leave off the .ts extension.

Overview

  • Open-source, released (as Angular 2+) 2016 by Google; a complete rewrite of the older AngularJS.
  • Written in TypeScript + HTML + CSS; needs Node.js to build/serve.
  • A full framework (routing, forms, HTTP, DI all included) — contrast with React, which is a library.

Anatomy of a component

💡 Three parts

Every component = an import, a @Component decorator (links template + styles to the class), and the class (your data + methods).

import { Component } from '@angular/core';

@Component({
  selector: 'app-home',                  // the HTML tag <app-home>
  templateUrl: './home.component.html',
  styleUrls: ['./home.component.css']
})
export class HomeComponent {
  counter: number = 0;
  name: string = "";
  count() { this.counter++; }
}
What this does, line by line
  • import { Component } from '@angular/core'; — brings in the @Component tool from Angular so you can use it below.
  • @Component({ ... }) — a decorator: it sits above the class and adds Angular configuration to it. Everything in the braces tells Angular how this component is wired up.
  • selector: 'app-home' — the custom HTML tag for this component; writing <app-home> anywhere drops this component in.
  • templateUrl / styleUrls — point to the HTML file (its visual layout) and the CSS file(s) (its styling). styleUrls is an array because a component can have several stylesheets.
  • export class HomeComponent { ... } — the actual class holding the component's data and logic. counter and name are data fields the template can show; count() is a method that adds 1 to counter (this means "this component instance").

Data binding — the four kinds

What this does, line by line
  • <button (click)="count()"> — event binding. The round brackets mean "when this event happens". So clicking the button runs the component's count() method (which adds 1 to counter).
  • {{ counter }} — interpolation: drops the current value of counter into the page as text. It updates automatically whenever counter changes.
  • <input [(ngModel)]="name"> — two-way binding. The [()] ("banana in a box") keeps the textbox and the name field in sync both ways: typing updates name, and changing name in code updates the textbox.
  • <p>Hi {{ name }}!</p> — shows whatever you typed live, because it reads the same name field.
⚠️ ngModel needs a module

[(ngModel)] only works if FormsModule is imported in your module / standalone component imports. Interpolation {{ }} is also called the "moustache" / "handlebars" syntax.

Conditionals: *ngIf and ng-template

The exam's Angular question expects you to show one button or another depending on state. The lecture style uses ng-template with [ngIf]:

<ng-template [ngIf]="counter > 5" [ngIfElse]="none">
  Clicking too much!
</ng-template>
<ng-template #none>Not clicking enough.</ng-template>

<!-- shorthand form (also valid) -->
<p *ngIf="counter > 5">Clicking too much!</p>
What this does, line by line
  • <ng-template [ngIf]="counter > 5" [ngIfElse]="none"> — only shows its contents if the condition counter > 5 is true. If it's false, Angular shows the template named by [ngIfElse] instead.
  • <ng-template #none>...</ng-template> — the #none is a template reference (a label). This is the "else" block that [ngIfElse]="none" pointed at.
  • <p *ngIf="counter > 5"> — the shorthand: the * does the same show/hide on a single element without writing a full ng-template. Use this when you don't need an else branch.

Lists: *ngFor

🎯 Highest-frequency Angular topic

Know *ngFor="let item of array" and how to read properties with item.property inside the loop.

// component
music = [
  { title: 'Track 1', rank: 500 },
  { title: 'Track 2', rank: 302 }
];

<!-- template -->
<div *ngFor="let m of music">
  <p>{{ m.title }} — rank {{ m.rank }}</p>
</div>
What this does, line by line
  • music = [ {…}, {…} ]; — in the component, an array of objects, each with a title and a rank.
  • <div *ngFor="let m of music"> — repeats this <div> once for every item in music. Each time round, m is the current object (first Track 1, then Track 2).
  • {{ m.title }} — rank {{ m.rank }} — inside the loop you read fields off the current item with a dot, so each repeat prints that item's own title and rank.

Dynamic styling: [ngStyle], [ngClass], [class.x]

🎯 "Use Angular CSS only" = use [ngStyle]

When the question says "do not use traditional CSS, only Angular CSS", it wants [ngStyle] with a conditional (ternary) expression — exactly the colour-by-count requirement in Q14.

<!-- inline style decided by a condition -->
<p [ngStyle]="{'color': count > 10 ? 'red' : 'green'}">Status</p>

<!-- toggle a single class -->
<p [class.active]="count > 10">Text</p>

<!-- multiple classes from a method -->
<p [ngClass]="setClasses()">Text</p>
What this does, line by line
  • [ngStyle]="{'color': count > 10 ? 'red' : 'green'}" — sets inline CSS from code. The part count > 10 ? 'red' : 'green' is a ternary ("if-else in one line"): if count is over 10 the colour is red, otherwise green. The text recolours by itself as count changes.
  • [class.active]="count > 10" — adds the CSS class active only while the condition is true, and removes it when false.
  • [ngClass]="setClasses()" — calls a component method that returns an object saying which classes should be on/off; use this when several classes depend on conditions.

Worked example — the 2021 cart (Q14)

Given a component with videogames (each {title, price, cart}), cartCount, and addToCart()/removeFromCart(), write the template:

<!-- grey when empty, lightgreen otherwise; show counts -->
<p [ngStyle]="{'color': cartCount === 0 ? 'grey' : 'lightgreen'}">
  Total: {{ videogames.length }} | In cart: {{ cartCount }}
</p>

<div *ngFor="let game of videogames">
  <p>{{ game.title }} — R{{ game.price }}</p>
  <ng-template [ngIf]="!game.cart">
    <button (click)="addToCart(game)">Add to Cart</button>
  </ng-template>
  <ng-template [ngIf]="game.cart">
    <button (click)="removeFromCart(game)">Remove from Cart!</button>
  </ng-template>
</div>
What this does, line by line (this is the full Q14 answer)
  1. <p [ngStyle]="{'color': cartCount === 0 ? 'grey' : 'lightgreen'}"> — colours this paragraph grey when the cart is empty (cartCount === 0) and lightgreen once it has items. This satisfies the "must use Angular CSS, not normal CSS" rule.
  2. Total: {{ videogames.length }} | In cart: {{ cartCount }} — prints how many games exist in total (.length of the array) and how many are in the cart.
  3. <div *ngFor="let game of videogames"> — loops once per game; game is the current one.
  4. <p>{{ game.title }} — R{{ game.price }}</p> — shows that game's title and price.
  5. <ng-template [ngIf]="!game.cart"> — the ! means "not". So this block shows only when the game is not in the cart, giving an "Add to Cart" button. Clicking it runs addToCart(game), passing the current game.
  6. <ng-template [ngIf]="game.cart"> — the opposite: when the game is in the cart, show "Remove from Cart!" which calls removeFromCart(game).

Net effect: each game shows exactly one button, the right one for its current state, and the counter paragraph recolours automatically.

⚠️ Mistakes to avoid (real script lost 7.5/10 here)
  • Writing Ionic tags (<ion-list>) for an Angular web question — read which framework is asked.
  • videogames.count — JS/TS arrays use .length, not .count.
  • Forgetting to pass the loop item: addToCart(game), not addToCart().
  • Using plain CSS / a style="..." attribute when told to use Angular CSS — use [ngStyle].

React

Meta's UI library (not a full framework). Component-based, state-driven, declarative, written in JSX.
W3Schools React Official React docs useState reference
📌 Version used in this module

React 18 / 19 with function components + Hooks (the slide teaches the useState hook). Class components (class X extends React.Component with this.state/setState) are the old pre-16.8 style — do not mix them in. Hooks (useState, useEffect) only work inside function components.

What is React?

  • A JavaScript library for building user interfaces, by Meta/Facebook.
  • Component-based (reusable pieces), state-driven (UI re-renders when state changes), declarative (describe what the UI is, not how to update the DOM).
  • Uses JSX — HTML-like syntax inside JavaScript. Each component returns one parent element.

Components & props (typed)

type MovieCardProps = { title: string; year: number; };

function MovieCard({ title, year }: MovieCardProps) {
  return (
    <div>
      <h2>{title}</h2>
      <p>{year}</p>
    </div>
  );
}

<MovieCard title="Inception" year={2010} />
What this does, line by line
  • type MovieCardProps = { title: string; year: number; }; — describes the props (inputs) this component accepts: a text title and a numeric year.
  • function MovieCard({ title, year }: MovieCardProps) { — a component is just a function. The { title, year } pulls those two props straight out of the object passed in, and : MovieCardProps type-checks them.
  • return ( <div>...</div> ); — returns the JSX (the visual output). Everything must sit inside one parent <div>.
  • {title} / {year} — the braces drop a JavaScript value into the markup, so the props show up on screen.
  • <MovieCard title="Inception" year={2010} /> — using the component: strings are passed in quotes, numbers in braces. Reusing it with different props gives different cards.
⚠️ JSX gotchas
  • JS values go in single braces: {title}. Strings can use quotes: title="Inception"; numbers need braces: year={2010}.
  • Use className, not class (class is a reserved JS word).
  • Return one root element — wrap siblings in a <div> or a fragment <>…</>.

State — the useState hook

import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState<number>(0);
  //     ^value   ^setter             ^type   ^initial value
  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
    </div>
  );
}
What this does, line by line
  • import { useState } from 'react'; — brings in the useState hook, React's way to give a component memory that survives re-renders.
  • const [count, setCount] = useState<number>(0); — creates a piece of state starting at 0. It hands back two things: count (the current value) and setCount (the only correct way to change it). <number> says the state is a number.
  • <p>Count: {count}</p> — shows the current value.
  • onClick={() => setCount(count + 1)} — when the button is clicked, call setCount with the new value (current + 1). Calling the setter tells React to re-run the component and repaint with the new number.
⚠️ Mistakes to avoid
  • Never mutate state directly (count++ or arr.push()). Always call the setter and pass a new value/array.
  • Calling a handler instead of passing it: onClick={handleClick} ✅ vs onClick={handleClick()} ❌ (the second runs on render).
  • Calling a hook conditionally or inside a loop — hooks must run at the top level of the component, every render.

What this does, line by line

  2. searchText = input.replace(" ", "+").toLowerCase(); — cleans the input in one chain: swap every space for +, then make it all lowercase (because the brief says getResults only works lowercase and dislikes spaces). The result is stored back into searchText.

  6. return true; — reached when something was found, telling Android the query was handled.
  7. The final return false; covers the "2 characters or fewer" case (nothing searched yet).

// Create — makes the collection if it doesn't exist
db.myCollection.insert({ name: "Bitcoin", price: 8521.32 });
db.myCollection.insert([ {...}, {...} ]);   // many at once

// Read
db.myCollection.find({});                       // all
db.myCollection.find({ name: "Bitcoin" });        // exact match
db.myCollection.find({ price: { $lt: 5200 } });   // $lt $lte $gt $gte $ne
db.myCollection.find({ $or: [ {name:"BTC"}, {name:"ETH"} ] });

// Sort (1=asc, -1=desc) and projection (1=show, 0=hide)
db.myCollection.find({}).sort({ price: -1 });
db.myCollection.find({}, { name: 1, price: 1 });

// Update — filter, $set, options
db.myCollection.update({ name: "Bitcoin" },
  { $set: { price: 8652.23 } }, { multi: false });

// Delete
db.myCollection.remove({ name: "Bitcoin" });  // one query
db.myCollection.remove({});                  // all docs
db.myCollection.drop();                      // whole collection
What each command does
  • insert — db.myCollection.insert({...}) adds one document (a record). Passing an array [ {...}, {...} ] adds several at once. If the collection doesn't exist yet, it's created.
  • find — find({}) with an empty filter returns everything; find({ name: "Bitcoin" }) returns only docs whose name equals "Bitcoin". The operators $lt/$lte/$gt/$gte/$ne mean less-than, ≤, greater-than, ≥, not-equal. $or matches if any listed condition is true.
  • sort — .sort({ price: -1 }) orders results; 1 is ascending (small→big), -1 is descending (big→small).
  • projection — the second argument to find chooses which fields come back: 1 = include, 0 = hide.
  • update — first the filter (which docs), then $set (the new values), then options. { multi: false } updates only the first match (true would update all).
  • remove — deletes docs matching the filter; an empty {} deletes all documents. drop deletes the entire collection (structure and all).

Aggregation pipeline — the big question

🎯 Q10 (9 marks): "average X per Y, sorted, written to a collection"

Map the words to stages: "per Y" → $group _id:"$Y"; "average/total/max" → $avg/$sum/$max; "only where…" → $match; "sorted highest→lowest" → $sort: {field: -1}; "write to collection Z" → $out: "Z".

StageDoes
$matchFilter docs (like SQL WHERE)
$groupGroup by _id; apply $sum/$avg/$min/$max
$sortOrder results (1 asc, -1 desc)
$outWrite the result into a new collection
 
// 2021 Q10 — average Age per Office, oldest→youngest, into AverageAge
db.CompanyData.aggregate([
  { $group: { _id: "$Office", AverageAge: { $avg: "$Age" } } },
  { $sort:  { AverageAge: -1 } },   // -1 = oldest first
  { $out:   "AverageAge" }
]);
What this does, stage by stage

aggregate([ ... ]) runs a pipeline: each { } is a stage, and its output feeds into the next, like a conveyor belt.

  1. { $group: { _id: "$Office", AverageAge: { $avg: "$Age" } } } — buckets all documents by their Office value (_id is the thing you group by). For each office it computes the average of the Age field and labels it AverageAge. The $ in "$Office"/"$Age" means "the value of this field".
  2. { $sort: { AverageAge: -1 } } — sorts those grouped results by the average age, -1 = descending, so oldest office first.
  3. { $out: "AverageAge" } — writes the final result into a collection called AverageAge (creating/overwriting it).
⚠️ Mistakes to avoid
  • Forgetting the $ prefix on the field being grouped/aggregated: $avg: "$Age", not "Age".
  • _id is mandatory in $group — it is the grouping key.
  • "Oldest to youngest" / "highest first" = descending = -1 (a very common direction slip).
  • Filtering on a computed total must come after the group, with $match (not in the original query).

Advanced Web Security

⚠️ Ethics & scope

These techniques are for understanding defences and for authorised testing only. Unauthorised testing is illegal. The exam wants you to identify attacks and prevent them.

1. Local File Inclusion (LFI)

User input is passed into include()/require() without sanitisation, letting an attacker read server files via path traversal.

// VULNERABLE PHP
$page = $_GET['page'];
include($page);          // attacker controls the included file

# Attack: ?page=../../../../etc/passwd   (path traversal)
# PHP wrapper: ?page=php://filter/convert.base64-encode/resource=index.php
What this does, line by line
  • $page = $_GET['page']; — reads a value straight from the URL query string (e.g. ?page=about.php) and trusts it blindly.
  • include($page); — pulls that file into the page. Because the attacker controls page, they decide which file gets loaded — that's the vulnerability.
  • ?page=../../../../etc/passwd — "path traversal": each ../ climbs up one folder, walking out of the website and into system files like the Linux password file.
  • The php://filter... line — a PHP "wrapper" trick that reads a source file and returns it base64-encoded, letting the attacker steal the site's own source code.

Defences: whitelist allowed page names; strip ../, null bytes and wrapper schemes; set open_basedir in php.ini to confine file access.

2. Remote File Inclusion (RFI)

Like LFI, but the included file is fetched from an attacker-controlled remote URL — almost always Remote Code Execution. Requires PHP's allow_url_include = On (off by default).

AspectLFIRFI
File sourceLocal filesystemRemote attacker server
PHP prerequisiteNone (just unsanitised include)allow_url_include = On

Defences: disable allow_url_include; never put user input in include(); whitelist; WAF.

3. Server-Side Request Forgery (SSRF)

An attacker makes the server issue requests to destinations of their choosing — including internal services the internet can't reach. (OWASP A10:2021.)

  • Cloud metadata theft: http://169.254.169.254/latest/meta-data/iam/security-credentials/ leaks AWS IAM keys → full account compromise.
  • Internal network scanning, port scanning, reaching admin panels protected only by network location.

Defences: validate/sanitise URLs, allowlist outbound domains, block loopback & RFC-1918 ranges, disable unused URL schemes, use AWS IMDSv2.

4. XML External Entity (XXE)

🎯 Q13 (2021): a portal parses user-uploaded XML and got compromised

Answer = XXE injection: a malicious DOCTYPE defines an external entity pointing at a server file (e.g. /etc/passwd); when parsed, its contents are echoed back, leaking sensitive data.

<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<user><name>&xxe;</name></user>   <!-- &xxe; expands to the file's contents -->
What this malicious XML does, line by line
  • <?xml version="1.0"?> — a normal XML declaration; nothing suspicious yet.
  • <!DOCTYPE foo [ ... ]> — a DOCTYPE block, which is allowed to define "entities" (reusable shortcuts). This is the door the attack walks through.
  • <!ENTITY xxe SYSTEM "file:///etc/passwd"> — defines an entity named xxe whose value is the contents of a file on the server. SYSTEM tells the parser to go fetch that external resource.
  • <name>&xxe;</name> — uses the entity. When a vulnerable parser expands &xxe;, it inserts the file's contents here, and if the app echoes the parsed data back, the secret file leaks to the attacker.
  • Fix: tell the XML parser to not process external entities/DTDs (or use JSON instead), and the whole trick stops working.

Attack variants: file disclosure, SSRF, blind/out-of-band exfil, DoS ("billion laughs" nested entities). Defences: disable external-entity (DTD) processing in the parser; prefer JSON; validate against a strict XSD; deploy a WAF.

class Car(models.Model):

    year_to = models.PositiveIntegerField(null=True, blank=True)  # optional
What this does, line by line
  • class Make(models.Model): — each model class becomes a database table; Make will be the "makes" table.
  • name = models.CharField(max_length=50, unique=True) — a text column up to 50 chars; unique=True means no two makes can share a name.
  • def __str__(self): return self.name — controls how a Make is shown as text (e.g. in the admin), displaying its name instead of "Make object".
  • make = models.ForeignKey(Make, on_delete=models.CASCADE) — links each Car to one Make (a relationship). CASCADE means if a Make is deleted, its Cars are deleted too.

  • (click)="count()" — when the button is clicked the browser fires a click event; Angular catches it and calls count() on the class.
  • [(ngModel)]="name" — combines [value]="name" (class→input) and (input)="name=$event.target.value" (input→class) into one shorthand. Typing in the box updates name, and any code change to name updates the box.

3 — Conditional rendering: *ngIf and ng-template

Angular gives you three ways to show/hide elements conditionally — know all three because the exam can ask for any.

<!-- Method 1: *ngIf shorthand on a single element (no else) -->
<p *ngIf="counter > 5">You are clicking too much!</p>

<!-- Method 2: ng-template with [ngIf] and an else branch -->
<ng-template [ngIf]="counter > 5" [ngIfElse]="notEnough">
  <p>Clicking too much!</p>
</ng-template>
<ng-template #notEnough>
  <p>Not clicking enough.</p>
</ng-template>

<!-- Method 3: *ngIf with else on the same line -->
<p *ngIf="counter > 5; else notEnough">Clicking too much!</p>
<ng-template #notEnough><p>Not clicking enough.</p></ng-template>
What this does
  • *ngIf="counter > 5" — Angular evaluates the expression. If truthy, the element (and all its children) are added to the DOM. If falsy, they are completely removed (not just hidden).
  • <ng-template> — a special Angular element that does not render any HTML by itself; it is a container for conditional/loop content.
  • [ngIfElse]="notEnough" — points to a template by its reference variable name (#notEnough). When the condition is false, Angular renders that template instead.
  • #notEnough — the hash declares a template reference variable: a local name for a template, used to point at it from [ngIfElse].
🎯 The 2021 exam expected ng-template style

When the question says "use templates", write two separate <ng-template [ngIf]="..."> blocks (one for true, one for false) rather than the shorthand. That is exactly the pattern in the 2021 Q14 answer.

9 — Services and dependency injection

A service is a class decorated with @Injectable() that holds reusable logic (usually API calls). You declare it once and inject it into any component that needs it.

// jokes.service.ts
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';

@Injectable()          // marks this class as injectable (can be injected into components)
export class JokesService {

  endpoint = 'https://v2.jokeapi.dev/joke/';

  // Angular provides an HttpClient instance via the constructor parameter
  constructor(private http: HttpClient) {}

  getRandomJoke(): Observable<any> {
    return this.http.get(this.endpoint + 'Programming?safe-mode&type=single');
  }
}
// jokes.component.ts — inject and use the service
import { HttpClientModule } from '@angular/common/http';
import { JokesService } from '../services/jokes.service';

@Component({
  selector: 'app-jokes',
  templateUrl: './jokes.component.html',
  providers: [JokesService],              // create a JokesService instance here
  imports: [HttpClientModule],            // needed for HttpClient to work
})
export class JokesComponent {
  jokes: any[] = [];

  // Angular DI hands us a JokesService instance
  constructor(private jokesService: JokesService) {}

  getRandom() {
    this.jokes = [];
    this.jokesService.getRandomJoke().subscribe(result => {
      this.jokes.push({ id: result.id, text: result.joke, category: result.category });
    });
  }
}
What this does, line by line
  • @Injectable() — marks the service so Angular's DI system knows it can be instantiated and injected.
  • constructor(private http: HttpClient) — Angular sees HttpClient as the type and injects a ready-made instance automatically. You never write new HttpClient().
  • http.get(url) — returns an Observable (not the data yet). Think of it as a promise that can fire multiple values. Nothing happens until you .subscribe().
  • providers: [JokesService] in @Component — tells Angular to create one instance of JokesService tied to this component's lifetime.
  • jokesService.getRandomJoke().subscribe(result => { ... }) — subscribes to the Observable. When the HTTP response arrives, Angular calls the callback with the data. result is the parsed JSON body.

10 — Reactive forms: FormBuilder, FormGroup, Validators

import { FormBuilder, FormGroup, Validators } from '@angular/forms';

export class SearchComponent {
  myForm: FormGroup;

  constructor(public formBuilder: FormBuilder) {
    // build the form — each field: [initialValue, validators]
    this.myForm = formBuilder.group({
      title: ["", Validators.compose([
        Validators.maxLength(30),              // max 30 characters
        Validators.pattern("[a-zA-Z ]*"),     // letters and spaces only
        Validators.required                    // must not be empty
      ])],
      email: ["", [Validators.required, Validators.email]],
      age:   [0,  [Validators.min(18), Validators.max(120)]]
    });
  }

  search() {
    if (this.myForm.valid) {
      const title = this.myForm.controls['title'].value;
      console.log('Searching for:', title);
    }
  }
}
<!-- HTML template -->
<form [formGroup]="myForm">
  <input formControlName="title"
         [class.invalid]="!myForm.controls['title'].valid">

  <p *ngIf="myForm.controls['title'].hasError('maxlength')">
    Title is too long.
  </p>
  <p *ngIf="myForm.controls['title'].hasError('required') && myForm.controls['title'].touched">
    Title is required.
  </p>

  <button [disabled]="!myForm.valid" (click)="search()">Search</button>
</form>
What this does, line by line
  • formBuilder.group({ ... }) — creates a FormGroup containing named FormControl objects. Each control has an initial value and a list of validators.
  • Validators.compose([...]) — combines multiple validators for one field into one. You can also pass an array directly: [Validators.required, Validators.email].
  • [formGroup]="myForm" on the form — connects the HTML <form> to the TypeScript FormGroup.
  • formControlName="title" on the input — connects this specific input to the title field in the group.
  • [class.invalid]="!myForm.controls['title'].valid" — adds the CSS class invalid when the field fails any validator (real-time feedback).
  • hasError('maxlength') — checks for a specific validator failure to show a precise error message.
  • this.myForm.valid — true only when all fields pass all their validators; use this to enable/disable the submit button.

CapacitorHttp — GET and POST

import { CapacitorHttp, HttpOptions, HttpResponse } from '@capacitor/core';

// ---- GET request ----
const doGet = async () => {
  const options: HttpOptions = {
    url: 'https://example.com/api/data',
    headers: { 'Content-Type': 'application/json' },
    params: { id: '123', limit: '10' }  // appended as ?id=123&limit=10
  };
  const response: HttpResponse = await CapacitorHttp.get(options);
  console.log(response.data);            // parsed JSON body
  console.log(response.status);         // HTTP status code (200, 404…)
};

// ---- POST request ----
async getData() {
  const options: HttpOptions = {
    url: 'https://wheatley.cs.up.ac.za/api/',
    headers: { 'Content-Type': 'application/json' },
    data: {                              // JSON body sent with the request
      studentnum: "u12345678",
      apikey:     "abc123",
      type:       "GetAllPlanes",
      limit:      10,
      return:     "*"
    }
  };
  const response: HttpResponse = await CapacitorHttp.post(options);
  this.planes = response.data.data;    // .data is the body; .data.data is the array
}
What this does, line by line
  • HttpOptions — a typed object defining the request. Must have at least url.
  • headers: { 'Content-Type': 'application/json' } — tells the server you're sending JSON. Required for POST requests that send a JSON body.
  • params (GET) — key-value pairs appended to the URL as a query string. They appear as ?key=value&key2=value2.
  • data (POST) — the JSON body. Passed as a plain JS object (not a string); Capacitor serialises it.
  • response.data — the parsed JSON body of the response. If the API wraps its payload in a data key, you access it as response.data.data.
  • await before the call — pauses the function until the HTTP response arrives. The function must be marked async.

7 — SearchView and onQueryTextChange — the 2021 Q7 question explained

Android's SearchView is a UI component that provides a text-search box. The method onQueryTextChange is called automatically every time the user types a character. You override it to implement predictive/live search.

<!-- 1. Declare a SearchView in the layout XML -->
<SearchView
    android:id="@+id/searchView"
    android:layout_width="match_parent"
    android:layout_height="wrap_content"
    android:queryHint="Search games..." />
// 2. In your Java Activity, set the OnQueryTextListener
SearchView searchView = (SearchView) findViewById(R.id.searchView);

searchView.setOnQueryTextListener(new SearchView.OnQueryTextListener() {

    // Called when user submits (presses Search button)
    @Override
    public boolean onQueryTextSubmit(String query) {
        getResults(query.toLowerCase(), false);
        return true;
    }

    // Called on EVERY keystroke — the predictive text method from the exam
    @Override
    public boolean onQueryTextChange(String input) {
        searchText = input.replace(" ", "+").toLowerCase();
        if (input.length() > 2) {
            if (getResults(searchText, false) == null) {
                if (getResults(searchText, true) == null) {
                    return false;
                }
            }
            return true;
        }
        return false;
    }
});
Walkthrough of onQueryTextChange (the 2021 Q7 answer)
  1. @Override — Java annotation meaning "this method overrides one from the interface/superclass". Required when implementing an interface method.
  2. public boolean onQueryTextChange(String input) — Android calls this with the full current text after every keystroke. Return true if you handled the query, false if not.
  3. searchText = input.replace(" ", "+").toLowerCase(); — two transformations in one chain: (a) replace every space with + because the hypothetical getResults function doesn't accept spaces; (b) lowercase because it only works with lowercase. The result is stored back into searchText — this is the key bit the real exam script got wrong by writing searchText.toLowerCase() without reassigning (String is immutable in Java).
  4. if (input.length() > 2) — don't search for 1–2 characters; that would trigger too often and return too many results.
  5. getResults(searchText, false) — first attempt: search without category expansion. It returns null when nothing found.
  6. If null, try getResults(searchText, true) — second attempt: expand to categories. If that is also null, return false (no match at all).
  7. If the first call succeeded (not null), the inner if is skipped and we fall through to return true.
⚠️ Java's String is immutable

myString.toLowerCase() does NOT change myString — it returns a new String. You must write myString = myString.toLowerCase(); to actually update the variable. This is different from JavaScript/TypeScript.

Mongo.js — complete annotated example

// INSERT — array of documents; each can have different fields
db.example.insert([
  { name: "Armand",  age: 21 },
  { name: "Katlego", age: 22 },
  { name: "Bob",     age: 30, married: true }  // extra field is fine — dynamic schema
])

// CRITERIA QUERY — one comparison operator
db.zipcodes.find({ pop: { $lt: 100 } })        // pop less than 100

// AND — two fields in the same object (implicit AND)
db.zipcodes.find({ city: "NEW SALEM" })                         // exact match
db.zipcodes.find({ city: "NEW SALEM", state: "PA" })            // both must match

// OR — explicit $or with an array of conditions
db.zipcodes.find({ $or: [{ city: "NEW SALEM" }, { state: "PA" }] })

// CRITERIA + PROJECTION + SORT (all chained)
db.zipcodes.find(
  { pop: { $lt: 100 }, pop: { $gt: 90 } }, // filter: 90 < pop < 100
  { _id: 0, loc: 0 }                        // projection: hide _id and loc fields
).sort({ pop: 1 })                           // sort ascending by pop

// UPDATE — $set changes specific fields without touching the rest
db.example.update(
  { name: "Armand" },           // 1st arg: which document(s) to change
  { $set: { age: 30 } }        // 2nd arg: what to change ($set is required — without it, the whole document is REPLACED)
)

// REMOVE — delete matching documents
db.example.remove({ name: "Armand" })   // removes docs matching the filter
db.example.drop()                         // deletes the ENTIRE collection (structure + all data)

// AGGREGATION PIPELINE — stages run in order, output feeds into the next
db.zipcodes.aggregate(
  { $group: { _id: "$state", pop: { $sum: "$pop" } } }, // group by state, sum populations
  { $match: { pop: { $lt: 1000000 } } },                  // keep only states with pop < 1M
  { $sort:  { pop: 1 } },                                  // ascending (smallest first)
  { $out:   "example" }                                    // write result into 'example' collection
)
Deep explanations of the trickier lines
  • { name:"Bob", age:30, married:true } — MongoDB's dynamic schema means documents in the same collection can have different fields. "Bob" has married but the others don't — totally valid.
  • { _id: 0, loc: 0 } — projection. 0 = exclude, 1 = include. You cannot mix 0s and 1s (except for _id which can be 0 alongside 1s). Hiding _id requires explicitly setting _id: 0.
  • $set in update — without $set, the second argument replaces the whole document with just those fields, deleting everything else. Always use $set unless you want a full replacement.
  • { $group: { _id: "$state", pop: { $sum: "$pop" } } } — _id is the field to group by (the $ prefix means "value of this field"). pop: { $sum: "$pop" } creates a new field called pop that sums all pop values in each group.
  • $match after $group — you must filter after the group because the field pop (the computed sum) didn't exist before the group stage ran.
  • $out: "example" — creates or overwrites the example collection with the pipeline results.
⚠️ Subtle bug in the example — know it

The combined criteria query { pop: {$lt:100}, pop: {$gt:90} } has two keys with the same name. In JavaScript, the second overwrites the first, so only $gt:90 is applied. The correct way to express 90 < pop < 100 is: { pop: { $gt: 90, $lt: 100 } } (both operators in the same object).
